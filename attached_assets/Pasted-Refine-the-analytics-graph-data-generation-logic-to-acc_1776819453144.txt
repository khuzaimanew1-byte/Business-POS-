### 🧠 CORE DATA PRINCIPLE (VERY IMPORTANT)

The graph must be **event-based**, not interval-based.

* A data point should be created **only when a real sale event occurs**
* Each time a cart is submitted (sale confirmed), generate a new data point
* There is **no fixed limit** on number of data points — it can be:

  * Many in a short time (high activity)
  * Very few (low activity)

---

### ⏱️ DAILY MODE — REAL-TIME EVENTS

In daily mode:

* Use **exact timestamps (hour + minute)** for each sale event
* Example:

  * If a product is sold at **5:27 PM**, create a data point exactly at that time
  * If multiple sales happen close together (e.g., 5:27, 5:29, 5:31), each should create its own point

Important:

* Do NOT group or average data into hourly buckets
* Do NOT reduce or limit data points
* Show all real events as they occurred

---

### 📆 WEEKLY / MONTHLY / YEARLY LOGIC

Apply the same event-based logic across all modes:

* Data points represent **actual sale events only**
* If no sale occurs on a specific day:

  * Do NOT create any data point for that day

Example:

* If on Friday no cart was submitted → **no point for Friday**
* If Saturday has multiple sales → multiple points on that day

However:

* The **timeline (x-axis)** should still reflect continuous time progression
* Missing days should appear as **gaps or flat sections**, not fake data

### 📉 NO FAKE OR GENERATED DATA

* Artificial points

Everything must be based on:
